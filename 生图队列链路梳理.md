# 生图队列链路梳理

## 1. 目标与范围

这份文档只讨论当前画布体系里的生成任务链路，重点是图片生成队列；视频链路与之共用大部分基础设施，因此一并标出差异点。

这里的“队列”不是后端真实队列实现本身，而是前端对生成任务的统一编排、持久化恢复、轮询、状态投影和展示。

核心问题包括：

1. 上游请求从哪里进入。
2. 任务何时算“提交中 / 已入队 / 生成中 / 整理中 / 失败”。
3. 超时、断网、后端异常分别在哪一层处理。
4. 前端如何把状态同步到元素、sessionStorage、队列面板和占位卡。
5. 页面刷新、面板关闭、提交中断后如何恢复。

## 2. 核心分层

### 2.1 面板 / 页面入口层

负责发起生成意图，不直接关心底层轮询细节。

- 图片生成器：src/components/lovart/ImageGeneratorPanel.tsx
- 分镜规划器宫格图：src/components/lovart/StoryboardPlannerPanel.tsx
- 分镜批量图片提交：src/app/canvas/page.tsx
- AI 编辑图片：src/app/canvas/page.tsx
- 视频生成器 / 分镜视频 / AI Designer 视频任务：同理走视频链路

这一层只做三件事：

1. 收集 prompt、模型、比例、参考图。
2. 通过 onSubmittingChange 或 onElementChange 先把“提交中”状态回写到页面。
3. 调用共享 flow，拿到 completed 或 pending 两类结果。

### 2.2 共享 flow 层

- 泛型内核：src/lib/generation-task-flow.ts
- 图片 flow：src/components/lovart/image-generation-flow.ts
- 视频 flow：src/components/lovart/video-generation-flow.ts

这一层统一了两件事：

1. 发起请求后，如何识别“同步返回结果”还是“只返回 taskId”。
2. 如果要等待结果，如何统一进入 waitForGenerationTask。

它不负责更新画布元素，也不负责显示 UI。它只负责把结果标准化成：

- completed: 已拿到 resultUrl
- pending: 只有 taskId，后续交给页面级轮询控制器

### 2.3 客户端传输层

- src/lib/ai-client.ts
- src/lib/direct-ai-client.ts

职责：

1. 统一请求 `/api/generate-image`、`/api/generate-video`。
2. 统一轮询 `/api/image-status`、`/api/video-status`。
3. 把后端状态规范化成 completed / failed / retryable-error / processing。

图片链路有一个额外分支：

- `preferDirect !== false` 时，先尝试浏览器直连 `directGenerateImage()`。
- 直连只要出现 CORS / 网络失败 / 超时，会自动回退到 Next.js 服务端代理。
- 这保证了“能直连就直连，不能直连也不阻断功能”。

### 2.4 服务端 API 代理层

- 图片提交：src/app/api/generate-image/route.ts
- 图片状态：src/app/api/image-status/route.ts
- 视频提交：src/app/api/generate-video/route.ts
- 视频状态：src/app/api/video-status/route.ts
- 共享错误与重试：src/app/api/_shared/ai-service.ts

职责：

1. 读取请求头里的 API Key / Base URL，或回退到服务端环境变量。
2. 做输入校验、参考图大小限制、上游请求重试、错误翻译。
3. 把上游原始返回映射成前端可消费的统一 JSON。

### 2.5 页面级轮询与队列投影层

- 轮询控制器：src/app/canvas/canvas-generation-controller.ts
- 队列投影：src/app/canvas/canvas-generation.ts
- 会话恢复：src/app/canvas/canvas-session-runtime.ts
- 提交 / 任务持久化：src/app/canvas/generation-persistence.ts

职责：

1. 对所有带 `generatingTaskId` 的生成器元素统一轮询。
2. 维护 `generatorSubmittingMap`、`generatingTaskId`、`generatingProgress`、`generatingError`。
3. 把原始元素状态投影成队列面板项和画布占位态。
4. 页面刷新后恢复未完成任务，或重新提交提交中断的任务。

## 3. 图片生成主链路

### 3.1 提交阶段

以图片生成器为例：

1. 面板调用 `onSubmittingChange(elementId, true, liveParams)`。
2. 页面里的 `handleGeneratorSubmittingChange()` 立即：
   - 把 `generatorSubmittingMap[elementId] = true`
   - 把面板当前 prompt / model / aspectRatio 等同步回元素，避免异步持久化滞后
   - 调用 `persistSubmission()` 写入 sessionStorage，记录“请求已经发起但还没拿到 taskId”
3. 面板调用 `runImageGenerationFlow()`。

此时前端 UI 已进入“提交中”，即使请求还没拿到 taskId，也已经有可恢复记录。

### 3.2 请求上游阶段

图片请求顺序如下：

1. `requestImageGeneration()` 先走 `resolveImageRequest()` 合并默认值。
2. 若 `preferDirect !== false`，先调用 `directGenerateImage()`：
   - 浏览器直连上游 `v1/images/generations`
   - 单次超时 30 秒
   - 网络/CORS/超时失败则返回 `null`，自动回退服务端代理
3. 回退或禁用直连时，调用 `/api/generate-image`
4. 服务端 `generate-image` 路由：
   - 校验 prompt
   - 处理参考图 URL / data URL / base64
   - 限制单张参考图最大 10MB，最多 6 张
   - 用 `fetchWithRetry()` 访问上游，当前配置为 3 次重试、每次 45 秒超时

视频链路类似，但：

- 不走浏览器直连
- 使用 `/api/generate-video`
- 单张参考图上限 15MB
- 图片数量限制跟模型有关

### 3.3 服务端返回给前端的两种结果

提交接口只会回两类有效结果：

1. `completed`
   - 已直接返回 `imageUrl` / `imageData` / `videoUrl`
2. `pending`
   - 只返回 `taskId`

图片路由若既拿不到 taskId、也拿不到直接结果，会返回 `status: unknown`，然后在 flow 层被判定为“未返回可用结果”。

### 3.4 前端接单后如何入队

共享 flow 返回 `pending` 后，调用方会：

1. 把元素写成 `generatingTaskId + generatingTaskType + generatingProgress=0`
2. 调用 `persistGeneration()` 把 taskId 写入 sessionStorage
3. 结束 `submitting` 状态，即 `generatorSubmittingMap[elementId] = false`

此时 UI 从“提交中”切到“排队中/生成中”，并由页面级轮询控制器接管。

## 4. 页面级轮询链路

### 4.1 谁会进入轮询

`getActiveGenerationTasks()` 当前只接管：

- `image-generator`
- `video-generator`

并要求：

- `generatingTaskId` 存在
- `generatingTaskId !== 'ai-editing'`

`storyboard-planner` 会进队列显示，但不被页面级轮询直接接管，它在组件内部通过 `waitForImageGenerationResult()` 恢复自己的宫格图任务。

### 4.2 轮询节奏

统一配置在 `GENERATION_POLLING_CONFIG`：

- 轮询间隔：1.5 秒
- 单次状态查询超时：15 秒
- 连续可重试错误阈值：3 次
- 图片无进度超时：12 分钟
- 视频无进度超时：20 分钟

这里有两层超时：

1. 浏览器请求 `/api/*-status` 的 15 秒超时
2. 服务端 `/api/*-status` 到上游的 15 秒超时

这次已经把第 2 层补齐，避免前端超时而服务端还一直挂住。

### 4.3 状态查询返回如何标准化

图片状态路由：

- 上游 `SUCCESS` -> `{ status: 'completed', imageUrl }`
- 上游 `FAILURE` -> `{ status: 'failed', error }`
- 其他状态 -> `{ status: 'processing', progress }`

视频状态路由：

- 上游 `SUCCESS` -> `{ status: 'completed', videoUrl }`
- 上游 `FAILURE` -> `{ status: 'failed', error }`
- 其他状态 -> `{ status: 'processing', progress }`

客户端 `pollGenerationTask()` 再进一步统一成：

- completed
- failed
- retryable-error
- processing

其中 `retryable-error` 的来源是：

- `/api/*-status` 返回非 2xx
- 这种错误不会立刻把任务判死，而是允许控制器继续重试

### 4.4 控制器如何决定失败

`useGenerationPollingController()` 维护每个任务的健康信息：

- `startedAt`
- `lastProgressAt`
- `lastProgress`
- `consecutiveErrors`

任务会被判失败的条件有三类：

1. 状态接口明确返回 `failed`
2. 连续 `retryable-error` 达到 3 次
3. 长时间没有任何进度推进，超过 `staleTimeoutMs`

一旦失败，统一走 `failGenerationTask()`：

- 清除内存里的健康信息
- `removeGeneration()`
- `clearSubmission()`
- `generatorSubmittingMap[elementId] = false`
- 把元素写成 `generatingError`
- 弹 toast

### 4.5 成功时如何落图

图片成功：

- `finalizePolledImageResult()` 负责把结果规范化进 ImageStore、修正尺寸、替换当前生成器元素

视频成功：

- `applyVideoGenerationSuccess()` 直接把元素从 `video-generator` 切成 `video`

两者都会：

- 清理 sessionStorage 里的 generation 记录
- 记录到项目媒体历史
- 发布完成提示

## 5. 队列状态机

当前前端显示层的状态机是一个投影状态机，不直接使用后端原始 status 文本，而是由元素字段推导。

### 5.1 元素级状态

1. idle
   - 没有 `generatingTaskId`
   - 不在 `generatorSubmittingMap`
   - 没有 `generatingError`

2. submitting
   - `generatorSubmittingMap[elementId] === true`
   - 但还没有 `generatingTaskId`

3. queued
   - 有 `generatingTaskId`
   - `progress <= 0`

4. running
   - 有 `generatingTaskId`
   - `0 < progress < 85`

5. finishing
   - 有 `generatingTaskId`
   - `progress >= 85`

6. failed
   - `generatingError` 存在

### 5.2 队列面板展示

队列数据由 `buildGenerationQueueItems()` 生成：

- 元素级任务：单个 image/video/storyboard-planner
- 分组任务：按 `generationBatchId` 聚合分镜批次

显示策略：

- `submitting`：提交中
- `queued`：排队中
- `running`：生成中
- `finishing`：整理中
- `failed`：失败

批量任务进度是批次成员的平均值，且对“已完成但已替换成 image/video 的元素”按 100 计分。

### 5.3 画布占位卡展示

`CanvasElementRenderer` 会根据：

- `generatingTaskId`
- `generatorSubmittingMap`
- `generatingError`

展示三种主文案：

1. 正在提交请求
2. 正在生成
3. 失败原因

这保证即使不打开左上角队列面板，用户也能从元素本身看见当前状态。

## 6. 提交中断与恢复机制

### 6.1 为什么要区分 submission 和 generation

当前持久化层刻意拆成两张表：

1. `persistSubmission()`
   - 还没拿到 taskId
   - 只知道 prompt/model 等提交参数

2. `persistGeneration()`
   - 已拿到 taskId
   - 可以继续轮询

这样才能区分两种恢复场景：

1. 页面关掉时请求刚发出去，但 taskId 还没回来
2. 页面关掉时已经拿到 taskId，只是轮询中断

### 6.2 已拿到 taskId 的恢复

项目加载时：

1. `snapshot.pendingGenerations` 从 sessionStorage 读出
2. 若数据库里的元素还没有 `generatingTaskId`，就把 taskId/progress 补回元素
3. 页面初始化后，轮询控制器会自动接管这些恢复出来的任务

### 6.3 还没拿到 taskId 的恢复

`canvas-session-runtime.ts` 会读取 `loadPendingSubmissions()`，对孤儿提交做恢复：

- 只处理 5 分钟内的记录
- 元素必须还存在且仍是 generator 类型
- 若元素已经自己有 taskId，则清掉 submission，不重复提交
- 若仍没有 taskId，则按原参数重新发请求

恢复结果有两种：

1. 直接拿到结果
   - 立即把 generator 替换成 image/video
2. 拿到新的 taskId
   - 回写元素并重新进入正常轮询

### 6.4 手动恢复

当前只有视频生成器暴露了手动输入 `task_id` 的恢复能力：

- `handleRecoverVideoTask()` 先单次查询状态
- 若已完成，直接拿结果落图
- 若还在处理中，回写 taskId，让自动轮询继续接管

图片暂时没有对外暴露等价的手动恢复 UI。

## 7. 失败与超时的分层处理

### 7.1 输入校验失败

发生在服务端路由最前面，典型返回：

- 400：缺 prompt、taskId、参考图数量非法
- 413：参考图过大

这种失败不会入队，面板会直接显示错误。

### 7.2 上游连接失败

发生在服务端访问 AI Gateway 时。

当前处理：

- 提交接口：`fetchWithRetry()` 3 次重试，每次 45 秒超时
- 状态接口：单次 15 秒超时
- 失败后统一转成 502 或 500，并给出 details

前端表现：

- 提交阶段失败：面板直接报错，不会进入轮询
- 轮询阶段失败：先记为 `retryable-error`，达到阈值后转失败

### 7.3 上游语义失败

例如：

- 内容安全拦截
- 额度不足
- 速率限制
- 模型无法出图

图片提交路由和直连客户端都做了常见错误翻译，最终前端能拿到中文可读信息。

### 7.4 状态查询异常

`pollGenerationTask()` 会把非 2xx 的状态查询转成 `retryable-error`。

控制器不会立刻失败，而是依赖两条红线：

1. 连续错误次数 >= 3
2. `lastProgressAt` 距当前时间超过无进度超时

### 7.5 完成但无结果

这是一类单独处理的异常：

- 状态接口说 completed
- 但拿不到 `imageUrl` / `videoUrl`

当前会直接进入失败分支，并给出“生成完成但未获取到资源链接”这类错误。

这能避免前端把“假完成”当成成功处理。

## 8. 现在这套链路的几个关键结论

### 8.1 现在的职责边界已经比较清晰

1. 面板 / page 入口只负责任务提交和结果落画布。
2. 共享 flow 只负责请求结果标准化。
3. ai-client 只负责传输与轮询协议标准化。
4. API 路由只负责代理、校验、重试、错误翻译。
5. 页面级 controller 才负责队列健康、超时、失败和 UI 投影。

### 8.2 当前最重要的状态源有三个

1. `generatorSubmittingMap`
   - 表示“请求已发起，但 taskId 还没回来”

2. 元素上的 `generatingTaskId / generatingProgress / generatingError`
   - 表示“这个元素当前对应什么任务、处于什么轮询状态”

3. sessionStorage 里的 submission / generation
   - 表示“页面刷新后怎么恢复”

这三者缺一不可。

### 8.3 这轮改动之后，两个明显缺口已经补齐

1. image/video 共享 flow 已收敛到统一泛型内核，不再镜像复制。
2. 服务端到上游的 timeout 已统一补上，前后端超时语义不再断层。

## 9. 仍然存在但可接受的剩余点

1. 图片没有像视频那样暴露手动 `task_id` 恢复入口。
2. 队列状态 `queued/running/finishing` 仍然是前端基于 progress 的投影规则，不是后端原始状态枚举。
3. `clearGeneratorError` 这类单字段 patch 还没有继续抽象成更细的 helper，但已经不影响主链路一致性。

## 10. 如果后续还要继续收口，建议优先级

1. 给图片生成器补一个和视频一致的手动 task_id 恢复入口。
2. 把“任务生命周期状态”抽成更显式的领域枚举，减少对 `progress <= 0 / <85 / >=85` 这种 UI 规则的依赖。
3. 若后端未来提供更稳定的状态码语义，可把 finishing / queued 的判定迁回协议层，而不是继续靠前端猜测。
