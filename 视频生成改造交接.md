# 视频生成改造交接

更新时间：2026-05-13

## 1. 本轮已完成内容

本轮核心目标不是视频，而是先把整套多平台图片链路打稳，避免后续视频接入时仍受单平台设置、模型选项和异步恢复问题影响。

已完成的关键改造：

- 新增独立平台 `MagicAPI / GeekNow`，没有硬塞进原有默认平台逻辑，而是和默认平台 `bltcy` 平行存在。
- 前后端 API 设置改造成按平台分别保存：
  - 当前平台 ID
  - 当前平台 Base URL
  - 当前平台 API Key
  - 各平台各自的本地默认配置
- 图片生成链路已支持 MagicAPI：
  - Gemini 原生生图接口
  - OpenAI 风格生图接口
  - 图片任务状态查询
  - 长耗时任务的本地异步桥接恢复
- 图片默认值已改成按平台隔离，不会再出现两个平台的模型和尺寸互相污染。
- 图片生成器、分镜生成器、画布内图片相关动作都已经切到“读取当前平台对应默认值”的方式。
- MagicAPI 的 `gpt-image-2` 和 `gpt-image-2-pro` 选项已经按 `main.py` 适配过多轮：
  - 新增 `21:9 -> 2240x960`
  - 新增 `9:21 -> 960x2240`
  - 把 `2560x1712` / `1712x2560` 的展示比例改成 `3:2` / `2:3`
- 设置中心层级问题已修复，设置弹窗现在会盖在生成器浮层上面。

## 2. 当前多平台架构

### 平台定义

- 默认平台：`bltcy`
- 新平台：`magicapi`

核心注册文件：

- `src/lib/ai-providers.ts`

这里定义了：

- 平台 ID / 标签 / 描述
- 默认 Base URL
- 允许的公开域名
- 对应环境变量名
- 平台支持的图片 / 视频 / 聊天模型列表

### 浏览器设置存储

核心文件：

- `src/lib/api-settings.ts`

现在请求头会带：

- `x-ai-provider`
- `x-ai-base-url`
- `x-ai-api-key`

服务端会按平台读取不同 env：

- 默认平台：`AI_API_KEY` / `AI_API_BASE_URL`
- MagicAPI：`MAGICAPI_API_KEY` / `MAGICAPI_API_BASE_URL`

### 运行时默认值

核心文件：

- `src/lib/workbench-settings.ts`
- `src/lib/generation-defaults.ts`

当前图片默认值按平台分离：

- `bltcy`
  - model: `gemini-3.1-flash-image-preview`
  - imageSize: `2K`
  - aspectRatio: `21:9`
  - quality: `auto`
  - generateCount: `1`
- `magicapi`
  - model: `gemini-3-pro-image-preview`
  - imageSize: `2K`
  - aspectRatio: `21:9`
  - quality: `auto`
  - generateCount: `1`

## 3. 图片链路现状

### 已打通的服务端路由

- `src/app/api/generate-image/route.ts`
- `src/app/api/image-status/route.ts`
- `src/app/api/_shared/ai-service.ts`
- `src/app/api/_shared/magicapi-image-jobs.ts`

### 当前图片侧的重要行为

- MagicAPI Gemini 原生生图走 `/v1beta/models/{model}:generateContent`
- MagicAPI OpenAI 风格生图走 `/v1/images/generations`
- 当上游任务很慢时，支持本地任务桥接，任务前缀为：
  - `magicapi-local:`
- 平台复制出来的图片 task 也支持前缀识别：
  - `magicapi:`
- 如果上游网关根本没开放图片 task 查询 URL，会给出明确失败提示，而不是假装还能恢复。

### 已确认的问题和处理方式

- 官方或网关的某些图片 task 查询地址会返回 `Invalid URL`
- 因此图片异步恢复不能完全依赖平台 task 查询接口
- 当前做法是：
  - 对本页面发起的长耗时图片任务，优先走本地桥接恢复
  - 对用户从平台后台复制回来的 task_id，只在网关真的支持查询时恢复
