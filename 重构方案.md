# OpenLovart 画布工程删减与重构方案

## 0. 结论先行

这个画布工程已经具备一部分工程化治理，不是完全失控的“不可维护屎山”。它有明确的领域模块、较多纯函数测试、Playwright 画布 QA、生成链路文档，以及已经抽出来的运行时控制器、持久化、轮询、图片 LOD、历史管理等模块。

但它确实符合不少“屎山代码”的典型特征：核心页面和核心 UI 面板过大，状态来源较多，副作用和业务编排混在 React 组件里，少数文件承担过多职责，回归风险主要集中在画布交互、生成任务恢复、项目持久化和媒体资产链路。

更准确的判断是：这是一个“已经被拆过一轮，但仍有巨型壳层和面板负债的复杂工作台工程”。如果继续往里堆功能，会快速滑向真正难改的屎山；如果按本方案做保功能重构，风险是可控的。

## 3. 是否符合“屎山代码”特征

### 3.1 符合的特征

1. 巨型入口文件

`src/app/canvas/page.tsx` 仍然是总线式页面。它已经调用了 `useCanvasDocumentState`、`useCanvasProjectPersistence`、`useCanvasSessionRuntime`、`useGenerationPollingController` 等抽出模块，但页面仍持有大量跨域状态和回调拼装。

典型风险：改一个生成功能可能牵到项目保存、媒体历史、图层高亮、队列面板、参考库回流。

1. 巨型面板组件

`LayersPanel.tsx`、`ImageGeneratorPanel.tsx`、`VideoGeneratorPanel.tsx`、`StoryboardPlannerPanel.tsx` 都是“状态 + 业务规则 + UI + 本地持久化 + 弹层菜单”的混合体。

典型风险：UI 结构调整会顺手碰到任务提交、历史恢复、mention 替换或存储逻辑。

1. 隐式状态来源多

画布状态来自多处：React state、refs、IndexedDB/localDb、localStorage、sessionStorage、URL query、元素字段、运行时 Map、worker 结果、Playwright 测试事件。

典型风险：某个 ref 更新不会触发 effect，某个 sessionStorage 恢复路径比 IndexedDB 保存更快，导致刷新/返回后状态不一致。

1. 副作用散布

全局事件、beforeunload、sessionStorage/localStorage、worker、fetch、object URL、图片 blob 存储、定时器和轮询都存在。部分已经集中，但仍有一些散在组件内部。

典型风险：组件卸载、刷新、并发生成、取消任务、拖拽过程中出现悬挂定时器、重复轮询、重复保存。

1. 类型模型过宽

`CanvasElement` 是宽接口，大量字段可选。虽然已经新增了 typed union 和 type guard，但很多调用点仍然使用宽 `CanvasElement`。

典型风险：某个字段只对 image-generator 合法，却可以被错误写进 image 或 frame；业务规则靠运行时约定维持。

1. 测试事件与产品逻辑耦合

`CanvasArea.tsx` 直接监听 `canvasTestEvents`，`LayersPanel.tsx` 也有测试事件入口。它们提升了 QA 能力，但也把测试后门混入产品组件。

典型风险：测试辅助行为无意改变真实交互，或者重构时误删导致 QA 失效。

1. Barrel 过渡层继续存在

`canvas-page-utils.ts` 现在是桶式导出，用来兼容旧导入。这对渐进重构有帮助，但长期会隐藏真实依赖方向。

典型风险：新代码继续从 barrel 拿所有东西，模块边界被重新抹平。

### 3.2 不符合“纯屎山”的反证

1. 有明确领域拆分

画布页已经拆出 `canvas-generation-controller.ts`、`generation-persistence.ts`、`canvas-session-runtime.ts`、`canvas-project-backflow.ts`、`canvas-element-factory.ts`、`canvas-image-assets.ts`、`use-image-finalizer.ts` 等模块。

1. 有大量纯函数测试

已覆盖 `canvas-render-plan`、`canvas-hit-test`、`canvas-snap-utils`、`canvas-types`、`history-manager`、`image-generation-models`、`lod-request-utils`、`workbench-settings` 等核心纯逻辑。

1. 有端到端画布 QA

`README.md` 和 `.github/workflows/canvas-qa.yml` 已把画布 QA 纳入推荐/CI 流程。

1. 生成链路已经文档化

`生图队列链路梳理.md` 对图片/视频生成、轮询、恢复、sessionStorage 持久化有清晰分层。

1. 最近已有重构成果

`graphify-out/REFACTOR_AGENT_BRIEF.md` 说明过往已经完成 worker 拆分、editor-kernel facade、generation-defaults、AI designer polling、typed-local-collection 等工作。

所以这不是“从零救火”，而是“继续沿已有拆分方向收口”。

## 4. 重构总原则

本方案要求“不改变现有功能”，因此原则是：

1. 不先改行为，只移动边界。
1. 先补测试再拆高风险逻辑。
1. 每次只拆一个职责，不顺手美化无关 UI。
1. 对已有测试事件和 `data-testid` 保持兼容。
1. 保留现有存储 key、元素字段、URL 参数和 public callback 形状。
1. 每个阶段结束都能跑 `typecheck + test + 对应 QA`。
1. 删除必须有证据：未引用、被新模块替代、或者只是兼容桶导出。

## 5. 可删减对象分级

### 5.1 可以优先删减的东西

这些删减通常不改变功能，但要先 grep / typecheck：

1. 过期注释和重复说明

一些文件顶部保留“从某处提取”的阶段性注释，可以在模块稳定后精简成职责说明。

1. Debug-only 输出

保留 `debugLog()`，逐步把散落的 `console.log` 迁入 `debugLog` 或删除。`console.warn/error` 保留在错误边界和 catch 里。

1. 重复 fallback 工具函数

凡是“判空、读 JSON、写 localStorage、事件订阅”的重复代码，统一迁到 `typed-local-collection.ts`、`project-asset-store.ts` 或专门 storage adapter。

1. 兼容 barrel 的新增使用

不立刻删除 `canvas-page-utils.ts`，但禁止新代码继续从它导入。后续逐步把旧导入改成领域文件直连，最后将 barrel 降到仅导出真正公共类型。

1. 重复的生成器 UI 片段

`ImageGeneratorPanel` 和 `VideoGeneratorPanel` 已共用 `generator-panel-sections.tsx`，还可以继续抽出模型菜单、尺寸/质量选择、恢复任务卡、引用素材栈的 view model。

1. 测试事件注册样板

保留能力，但把注册逻辑从组件主体抽到 `useCanvasTestEventBridge` / `useLayersPanelTestBridge`，生产逻辑不直接读测试事件常量。

### 5.2 暂不建议删除的东西

1. `canvas-page-utils.ts`

它是兼容层，不是理想结构，但贸然删除会制造大量机械改动。应该作为阶段末清理。

1. sessionStorage 生成恢复逻辑

它解决的是 IndexedDB 异步保存赶不上页面离开的竞态，不能因为“看起来重复”就删。

1. 画布 QA 脚本里的等待和截图

脚本偏长，但它们是重构保护网，先不要减。

1. `data-testid`

现在 QA 严重依赖这些标记。重构阶段不要删，除非同步改 QA。

1. 宽 `CanvasElement` 接口

可以逐步用 union 收紧调用点，但不能一次性拆成数据库 schema 级别的多表/多类型存储，否则迁移风险太高。

## 6. 目标架构

### 6.1 画布页面目标

`src/app/canvas/page.tsx` 最终只做五件事：

1. 组合 page-level hooks。
1. 组装传给 `CanvasArea` 和面板的 domain ports。
1. 渲染页面级面板和 overlay。
1. 处理极少数跨域 orchestration，例如“生成完成后写入媒体历史并高亮”。
1. 挂接 Next.js route/search params。

目标行数：从 4,460 行降到 1,500 至 2,200 行。不是追求短，而是减少跨域细节。

### 6.2 CanvasArea 目标

`CanvasArea` 最终是交互 shell，而不是所有交互实现：

- 指针交互继续放 hook。
- 渲染计划继续纯函数化。
- 右键菜单已经独立，继续保持。
- 测试事件桥移出。
- 对齐/布局动作移入独立 hook。
- 媒体预览/播放 overlay 保持组件化。

目标行数：从 1,554 行降到 900 至 1,100 行。

### 6.3 生成器面板目标

图片/视频/分镜生成器共享三层：

1. `generator-panel-state`：从元素读取初始状态，保存状态到元素，处理恢复。
1. `generator-panel-view-model`：模型选项、可用尺寸、质量、引用素材 preview、mention suggestions。
1. `generator-panel-sections`：无业务副作用的 UI 片段。

目标：图片和视频面板主体各降到 800 至 1,100 行，复杂业务放 hook 或纯函数。

### 6.4 图层面板目标

`LayersPanel` 拆成：

- `layers-tree-model.ts`：树构建、flatten、filter、sort、虚拟窗口计算。
- `layers-dnd-model.ts`：拖拽 payload、drop target、reorder/move parent 参数归一化。
- `layers-storyboard-meta.ts`：分镜字段校验、模板保存/删除、audit 状态。
- `LayerRow.tsx`：单行渲染。
- `LayersPanel.tsx`：只组合 state 和子组件。

目标：主体降到 700 至 1,000 行。

### 6.5 类型目标

继续保留 `CanvasElement` 宽接口作为持久化兼容层，但新增业务代码优先使用：

- `CanvasImageElement`
- `CanvasVideoElement`
- `CanvasFrameElement`
- `CanvasGeneratorElement`
- `CanvasMediaElement`
- `isCanvasElementOfType()` 等 guard

目标：减少“某字段只对某类型有效但任意地方都能写”的隐式风险。

## 7. 分阶段实施方案

### 阶段 A：建立重构基线

目的：保证后续每一步都有可比对的回归线。

动作：

1. 记录当前最大文件行数和测试基线。
1. 跑：
    - `npm run typecheck`
    - `npm run test`
    - `node scripts/verify_lod_regressions.mjs`
    - `npm run qa:canvas:scenarios`
    - `npm run qa:canvas:gpt-mentions`
1. 把每次重构前后的行数和测试结果写入一个轻量记录，例如 `artifacts/refactor-baseline/summary.md`。

不改功能代码。

验收：所有基线命令通过，形成可追溯 baseline。

### 阶段 B：禁止新增依赖回流

目的：先止血，防止拆出来的模块又被 page.tsx 或 CanvasArea 吸回去。

动作：

1. 约定新代码不得新增 `from './canvas-page-utils'`，必须从实际领域模块导入。
1. 约定组件不得直接新增 localStorage/sessionStorage 写入，除非放在 storage adapter。
1. 约定生成器面板不得直接读 `workbench-settings.ts`，继续走 `generation-defaults.ts`。
1. 约定 UI 调整不得删除现有 `data-testid`。

可选自动化：加一个轻量脚本检查新增导入和敏感模式。

执行进度：`canvas-page-utils` 依赖回流自动化已落地。新增 `scripts/check_canvas_boundaries.mjs` 和 `npm run check:canvas-boundaries`，禁止 `src/app/canvas` 内继续新增 `from './canvas-page-utils'` 导入，要求新代码从实际领域模块直连。

验收：不改变运行行为，但 code review 有明确边界规则。

### 阶段 C：拆 `LayersPanel.tsx`

优先级高，因为它体积最大且纯逻辑较多，适合保功能拆分。

建议拆分顺序：

1. 把 `LayerTreeBuilder` 移到 `layers-tree-model.ts`。
1. 给 `buildTree`、`flattenRows`、`filterRows`、排序逻辑补单元测试。
1. 把拖拽事件 detail 解析移到 `layers-dnd-model.ts`。
1. 把分镜元数据校验与模板逻辑移到 `layers-storyboard-meta.ts`。
1. 抽 `LayerRow.tsx`，保持 props 明确，不在 row 内直接碰全局状态。

不要做：

- 不重写图层 UI。
- 不改拖拽行为。
- 不改分镜字段含义。

验证：

- `npm run test -- src/components/lovart/layers-tree-model.test.ts`
- `npm run typecheck`

## 12. 不建议的重构方式

1. 一次性重写画布。

画布交互、持久化和生成恢复耦合到大量真实用户工作流，重写成本远高于收益。

1. 先追求“文件短”。

如果只是把函数搬到更多文件，但依赖方向不清晰，会从“单文件屎山”变成“多文件迷宫”。

1. 把所有状态塞进全局 store。

现在的问题不是缺全局 store，而是状态 owner 不够清晰。直接上全局 store 可能扩大副作用范围。

1. 删除 QA 脚本来减少仓库体积。

QA 是重构保护网，不能删。脚本可以以后再分层整理。

1. 同时改 UI 和底层行为。

视觉调整和行为重构必须拆开，否则回归定位困难。

## 14. 最终建议

这个工程最需要的不是“推倒重来”，而是继续做有验收的小步拆分。当前最应该优先处理的是：

1. `LayersPanel.tsx`，因为大且纯逻辑多，收益最高、风险相对低。
1. `ImageGeneratorPanel.tsx` / `VideoGeneratorPanel.tsx`，因为后续产品需求一定会继续加到生成器上。
1. `CanvasArea.tsx` 的测试桥和画板动作，因为可以减少核心交互壳层噪音。
1. `page.tsx` 的项目参考、分镜、工具动作，因为这是页面总线继续变薄的关键。

如果按这个顺序推进，既能保住现有功能，也能让后续开发从“怕碰”变成“知道该碰哪里”。
